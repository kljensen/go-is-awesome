The Go language is

  • compiled
  • statically typed
  • garbage-collected

  • lightweight, avoids unnecessary repetition
  • object oriented, but not in the usual way
  • concurrent, in a way that keeps you sane
  • designed for working programmers

http://talks.golang.org/2012/simple.slide#5

Go history

  • Conceived in 2007 at Google
  • Profoundly experienced authors
    • Ken Thompson: Unix, B language (predecessor to C), Plan 9, UTF–8
    • Rob Pike: Plan 9, Inferno OS, Limbo language, UTF–8
    • Russ Cox: libtask for Unix
  • Announcement in November 2009.
  • Go 1 released in March 2012.

Design

Consensus drove the design. Nothing went into the language until [Ken Thompson, Robert Griesemer, and myself] all agreed that it was right. Some features didn’t get resolved until after a year or more of discussion.

- Rob Pike

Iron.io

How We Went from 30 Servers to 2: Go
Our sustained CPU usage on our servers was approximately 50-60%... At some threshold above 50%, our Rails servers would spike up to 100% CPU usage and become unresponsive.

After we rolled out our Go version, we reduced our server count to two and we really only had two for redundancy. They were barely utilized, it was almost as if nothing was running on them. Our CPU utilization was less than 5% and the entire process started up with only a few hundred KB's of memory (on startup) vs our Rails apps which were ~50MB (on startup).
http://blog.iron.io/2013/03/how-we-went-from-30-servers-to-2-go.html

Goroutines

Independent threads of execution

Goroutine details

  • Like "&" at the end of a shell command
  • Lightweight, can have 10k+ no sweat
  • Each gets its own stack that grows, shrinks
  • Are multiplexed onto OS threads by runtime

Channels

Used for communicating, usually between goroutines

Buffered Channels

Channel details

Don't communicate by sharing memory; share memory by communicating.
  • <-c blocks if nothing to read
  • c<- blocks if channel is full
  • Powerful and expressive synchronization mechanism.

Select

Select details

  • A control structure unique to concurrency
  • Like "switch", but each case is a communication
  • Blocks until one can proceed
